The Onion Router (Tor)

Overview:
Tor specifically stands for The Onion Router. It is an implementation of onion routing which was developed in the mid 90s by United States Naval research. Generally, onion routing is "bouncing" a connection between different routers so that they’re hard to track, providing anonymity. Most of the time, we encrypt messages so that people can’t read a message we sent even if they have seen that we’ve sent it. Sometimes, however, we don’t want people to see that we’ve sent a message at all.

How It Works:
Say that a user wants to communicate with a server (i.e. a Facebook user wants to view their profile). Normally, the user makes a direct connection to the server to communicate with it. 

This user however, wants to remain anonymous. Not only does the user not want anyone to read the data being sent and recieved, the user also does not want anyone to see that they are communicating with that server at all. If the user signs into the server, this obviously defeats the purpose of using Tor because that reveals who the user is.

Onion routers “bounce” the user’s connection through other computers or servers called nodes. The last “bounce” is the node that will visit the server on the user’s behalf, similar to a proxy. Onion routers use multiple levels of encryption to confuse people who are trying to understand what is happening. In this example, Tor’s implementation of onion routing will use three intermediate nodes. For the user to communicate with the server, the user sends the message to node 1, who sends it to node 2, who sends it to node 3, who sends it to the server. For the server to respond, a new message is sent back the way it came. Without any special encryption, this network does not do anything except increase latency, the speed at which data is sent and recieved between the user and the server. 

What’s clever about onion routing is that no one in this network knows anything about the whole connection: they only know what is immediately before them and what is immediately after them.
There are three different encryption keys (K1, K2, and K3 in the diagram) automatically generated by the user, most likely AES keys. Node 1 knows K1, Node 2 knows K2, and Node 3 knows K3.  

The users sends one message, encrypted in layers with different keys for each layer.

As seen above, the message is first sent with all layers of encryption, and each node has the key to unlock the outermost layer. Once the node takes off the layer of encryption, it sends the still encrypted message to the next node. Only after the message has reached the final node is it fully decrypted and readable. On the way back, the same process happens in reverse. The message is received by node 3 from the server, encrypted with the K3 key, and sent to node 2, etc. Only the user has all the keys to decrypt all the layers and read the response.
The really interesting property of this network is that node 2 only needs to know the address of node 1, and node 3. It does not know who the user is, what the messages that it is sending and receiving mean, or who the server is. All node 2 knows is that when node 1 sends an encrypted message, it decrypts a layer with its K2 key, and sends it to node 3 (and the reverse process from receiving messages from node 3).
Security:
This system is strong because of the layers of encryption and the number of nodes. If a node was compromised by an attacker, all they see is that the computer is acting as an onion routing node with two other computers (on on each side). However, they can’t interpret the message at any of the nodes discovered.
In the example above, Tor is using 3 nodes for simplicity of the example. Using the Tor protocol, no one knows how many nodes there are (i.e. there could be hundreds of nodes and layers).
Creating The Circuit:
The network connecting the client (user’s machine in the example above), the nodes, and the server is referred to as a circuit. Tor’s documentation specifies how a client establishes the keys, how the messages are sent, how each node should decrypt the message, and how the node should forward the message on to the next node.

The client creates the circuit by first establishing a shared key with the first node, then instructing the first node to do the same with the next node, until the last node establishes a connection with the server. The circuit is extended every time each node wants to add new nodes on the end before the connection is established with the server. Every time this happens, all the previous messages have an additional layer of encryption, strengthening the security of the message. 

Latency:
The speed at which data is sent between the client and the server is significantly reduced every time a node is added to the circuit. Each individual node could also be serving a lot of clients at once, serving as nodes in multiple circuits. Each node in the circuit does not have to be located very close together; a Tor circuit could be global. Using an onion router definitely increases the time a client has to wait to send and receive data, but the benefit is that no one can see that you're communicating with the server at all.
Cells:
Tor messages are called cells, and cells are all 512 bytes long. The way that a router or a node tells whether to send a message to another node or to read the message is basically on a small hash or digest that’s held in the message. It looks at the tail end of the message, and calculates a hash on it. If the hash matches the hash the node has, it reads the message. If it does not, it forwards it on. After all this is done, the cell size is still 512 bytes long. This is important because if the size of the cell increases when it has more layers of encryption and decreases when it has less layers, it would be very easy to understand the circuit.
It's Not Perfect:
There is no such thing as “complete anonymity.” While onion routing does add a great amount of security to a user’s connection to a server, attackers can still find ways to expose the user’s connection to the server.
If the user signs in to the server using an email and password (i.e. the user signs in to their Facebook account), then the user is no longer anonymous because the server (i.e. Facebook) knows who is connected to them.
If the server that the client is connecting to is not encrypted at all, then the message that is sent from the last node to the server is sent can be intercepted and read. For this reason, it is still important to use TLS or HTTPS connections to the server even if the client is onion routing.
If someone is intercepting messages at point A (the front of the network) and point B (the exit node), they can figure out then entire picture, which is the one “achilles heel” of onion routing.

How To Use Tor:
To use onion routing, you can install the Tor browser on your machine, which functions as a web browser. Tor can also be used as a service to pipe other services and connections through it. Tor is not illegal because it is just encryption.
